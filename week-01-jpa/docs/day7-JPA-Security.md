### 1. 필터-트랜잭션 경계와 OSIV (Open Session In View)

**정의 및 발생 원인**
Spring Security의 필터 체인은 서블릿 컨테이너 영역에 위치하며, Spring MVC의 `@Transactional`이 관리하는 서비스(Service) 계층보다 앞서 실행됩니다. 데이터베이스 연결을 관리하는 영속성 컨텍스트(Persistence Context)의 생명주기는 OSIV 설정에 따라 결정됩니다.

**OSIV 설정에 따른 시스템 동작 비교**

* **OSIV ON (Spring Boot 기본값)**:
* 동작: 클라이언트의 요청이 필터를 통과할 때부터 응답이 나갈 때까지 영속성 컨텍스트와 데이터베이스 커넥션을 유지합니다.
* 문제점: 필터나 컨트롤러에서 외부 API 호출 등 대기 시간이 발생하면 커넥션을 점유한 채로 대기하므로, 트래픽이 집중될 경우 데이터베이스 커넥션 풀(Connection Pool) 고갈로 인한 장애가 발생합니다.


* **OSIV OFF (실무 권장)**:
* 동작: `@Transactional`이 선언된 서비스 계층 내에서만 커넥션을 유지하고 반환합니다.
* 문제점: 서비스 계층을 벗어난 보안 필터에서 지연 로딩(Lazy Loading)으로 연관 데이터에 접근하려 하면 데이터베이스 커넥션이 없으므로 `LazyInitializationException`이 발생합니다.



### 2. JPA N+1 문제와 Fetch Join 최적화

**정의**
JPA에서 지연 로딩으로 설정된 연관관계를 조회할 때, 최초 조회 쿼리(1) 외에 연관된 데이터 개수(N)만큼 추가적인 데이터베이스 쿼리가 발생하는 현상입니다.

**보안 필터에서의 발생 시나리오 및 해결책**

* **상황**: JWT 필터에서 토큰 파싱 후 데이터베이스의 유저 정보와 권한(Role) 데이터를 대조해야 합니다.
* **문제 발생 (일반 조회)**: `UserRepository.findByEmail()` 호출 시 유저 정보만 가져오며, 필터에서 `User.getRoles()`를 호출하는 순간 권한 개수만큼 쿼리가 추가로 발생합니다. OSIV가 OFF 상태라면 이 과정에서 예외가 발생하여 시스템이 정지합니다.
* **해결 (Fetch Join)**:
* JPQL `JOIN FETCH` 또는 `@EntityGraph`를 사용하여 쿼리를 작성합니다.
* 효과: 단일 SQL JOIN 쿼리로 유저 정보와 연관된 권한 데이터를 한 번에 메모리(영속성 컨텍스트)에 로드합니다. 이를 통해 필터 계층에서 데이터베이스 추가 연결 없이 안전하게 권한 데이터에 접근할 수 있으며 조회 성능이 향상됩니다.


### 1. 유저 정보를 조회하는 시점

정확히 **JWT 신분증의 유효성(서명 검증, 만료일 확인) 검사가 끝난 직후**입니다.
`JwtAuthenticationFilter` 내에서의 상세 흐름은 다음과 같습니다.

1. 클라이언트가 보낸 HTTP 요청 헤더에서 토큰 문자열을 추출합니다.
2. 토큰이 변조되지 않았는지, 만료되지 않았는지 검증합니다 (`jwtTokenProvider.validateToken(token)`).
3. 검증을 통과하면 토큰의 페이로드에서 유저 식별자(이메일)를 꺼냅니다 (`jwtTokenProvider.getSubject(token)`).
4. **[DB 접근]** 추출한 이메일을 키로 삼아 DB에서 유저 정보를 조회합니다 (`customUserDetailsService.loadUserByUsername(email)`).

### 2. 서비스 계층 밖에서의 DB 접근 가능 여부

**서비스 계층(`@Transactional`) 밖인 보안 필터에서도 데이터베이스 조회는 얼마든지 가능합니다.** 서비스 계층 밖이라고 해서 DB 접근 자체가 물리적으로 막히는 것은 아닙니다.

스프링 데이터 JPA의 리포지토리 메서드(`findByEmailWithRoles`)는 트랜잭션이 없는 곳에서 호출되더라도, 내부적으로 데이터베이스 커넥션 풀(HikariCP)에서 커넥션을 일시적으로 빌려와 조회를 수행한 뒤 즉시 커넥션을 반환하도록 설계되어 있습니다.

### 3. 그렇다면 무엇이 문제이고, 왜 Fetch Join을 쓰는가?

문제는 조회가 아니라, **조회해 온 엔티티의 연관 데이터를 필터 계층에서 '추가로' 가져오려고 할 때(지연 로딩)** 발생합니다.

* **일반 조회 시나리오 (에러 발생)**:
1. 필터에서 `UserRepository.findByEmail()` 호출 -> 일시적으로 커넥션을 얻어 `users` 테이블만 조회한 후 커넥션을 즉시 반환합니다.
2. 필터에서 스프링 시큐리티 인증 객체를 만들기 위해 `user.getRoles()`를 호출하여 권한 목록에 접근을 시도합니다.
3. 권한(`roles`) 데이터는 `LAZY`(지연 로딩)로 설정되어 있으므로, JPA는 DB에 추가 쿼리를 날려야만 합니다.
4. 하지만 이곳은 서비스 계층 밖이며 OSIV가 OFF 상태이므로, **해당 요청을 처리할 영속성 컨텍스트와 DB 커넥션이 이미 닫혀 있는 상태**입니다.
5. 결과: 추가 조회가 불가능하므로 `LazyInitializationException` 예외가 발생하며 시스템이 다운됩니다.


* **Fetch Join 적용 시나리오 (해결)**:
1. 필터에서 `UserRepository.findByEmailWithRoles()` 호출 -> 일시적으로 커넥션을 얻어 **`users` 테이블과 연관된 `user_roles` 테이블을 단 한 번의 SQL JOIN 쿼리로 묶어서 통째로 조회**합니다.
2. 조회 결과 데이터(유저 정보 + 권한 목록)가 처음부터 전부 자바 메모리 상의 객체에 로드됩니다.
3. 필터에서 `user.getRoles()`를 호출하더라도, **이미 메모리에 데이터가 존재하므로 DB에 다시 접근할 필요가 없습니다.**
4. 결과: 서비스 계층 밖이더라도 예외 없이 안전하게 인증 처리가 완료됩니다.


---

이 요청 생명주기(Request Lifecycle) 상에서 OSIV 설정과 JPA 조회 방식에 따라 DB 커넥션과 영속성 컨텍스트의 상태가 어떻게 변화하는지 단계별로 확인할 수 있는 시뮬레이터입니다.

### 3. 통합 테스트 스레드 격리와 MockMvc

**정의**
스프링 부트 환경에서 통합 테스트 시 데이터베이스 변경 사항을 원래 상태로 복구(Rollback)하기 위해 사용하는 메커니즘과 서버 구동 방식 간의 충돌 문제입니다.

**테스트 환경에 따른 롤백(Rollback) 동작 비교**

* **실제 웹 환경 (`WebEnvironment.RANDOM_PORT`)**:
* 동작: 내장 톰캣 서버가 백그라운드의 독립적인 스레드에서 구동됩니다.
* 문제점: 테스트를 실행하는 스레드와 실제 웹 요청을 처리하는 톰캣 스레드가 분리되어 있습니다. 따라서 테스트 코드에 `@Transactional`을 선언하더라도, 톰캣 스레드에서 발생한 트랜잭션 커밋(Commit)에는 영향을 주지 못해 데이터베이스 롤백이 실패하고 데이터 오염이 발생합니다.


* **가상 웹 환경 (MockMvc)**:
* 동작: 실제 톰캣 서버를 구동하지 않고 서블릿 컨테이너의 동작을 시뮬레이션하여 단일 스레드 내에서 요청과 응답을 처리합니다.
* 효과: 테스트 코드와 웹 처리 로직이 동일한 스레드를 공유하므로, `@Transactional`에 의한 롤백 메커니즘이 데이터베이스 계층까지 정상적으로 전파되어 상태 무결성을 유지합니다.
