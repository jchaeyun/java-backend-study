
## 1. 프록시(Proxy)와 지연 로딩(Lazy Loading)

### **프록시(Proxy)의 정체**

* **개념:** 실제 객체 대신 하이버네이트가 생성한 **가짜 대역 배우**입니다.
* **구조:** 실제 엔티티 클래스를 상속(Inheritance)받아 생성되므로 겉모양은 똑같습니다.
* **특징:**
* 내부에 실제 객체의 주소값을 담는 **`target`** 포인터를 가집니다.
* 식별자(ID)는 태어날 때부터 가지고 있어, ID를 조회할 때는 DB에 가지 않습니다.
* 실제 데이터가 필요한 순간(메서드 호출)에만 DB를 조회합니다.



### **지연 로딩(Lazy Loading)**

* **정의:** 연관된 엔티티를 즉시 가져오지 않고, 실제 사용할 때까지 조회를 미루는 기술입니다.
* **설정:** `@ManyToOne(fetch = FetchType.LAZY)`
* **장점:** 불필요한 JOIN이나 추가 쿼리를 방지하여 네트워크 부하를 줄입니다. 실무에서는 **무조건 LAZY**를 기본으로 사용합니다.

### LazyInitializationException의 원인
프록시의 동작 원리를 알면 원인이 명확해집니다.
**"이미 퇴근한 매니저(영속성 컨텍스트)에게 일을 시켜서"** 발생
* **상황**: 트랜잭션이 종료(commit 또는 close)되면 영속성 컨텍스트도 함께 종료됩니다.
* **문제**: 트랜잭션 밖에서(예: Controller나 View 레이어) 아직 초기화되지 않은 프록시의 메서드(item.getName())를 호출하면 어떻게 될까요?
* **결과**: 프록시는 영속성 컨텍스트에 "DB 좀 다녀와 줘"라고 요청하지만, 영속성 컨텍스트가 이미 죽어있기 때문에 도움을 줄 수 없습니다. 이때 발생하는 것이 LazyInitializationException입니다.

## 2. 핵심 동작 원리 (사용자 질문 답변 포함)

### **A. `getClass().getName()` vs `getName()**`

왜 전자는 쿼리가 안 나가고 후자는 나가는가?

* **`getClass().getName()` (껍데기):** 자바 객체 자체의 타입 정보(클래스 명)를 묻는 것입니다. 프록시 객체는 이미 메모리에 본인의 클래스 정보를 들고 있으므로 DB에 갈 필요가 없습니다.
* **`getName()` (알맹이):** 엔티티 내부에 저장된 실제 데이터 필드 값을 요구하는 것입니다. 프록시는 껍데기일 뿐 데이터를 갖고 있지 않으므로, 이때 비로소 프록시 초기화(DB 조회)가 일어납니다.

### **B. `IDENTITY` 전략과 `persist()`의 예외**

* **일반 원칙:** `persist()`는 1차 캐시에 저장만 하고 쿼리는 나중에 보냅니다.
* **IDENTITY 예외:** MySQL의 `AUTO_INCREMENT`처럼 DB가 ID를 정해주는 방식은, 데이터를 넣기 전엔 ID를 알 수 없습니다. JPA는 1차 캐시에 넣기 위해 ID(Key)가 꼭 필요하므로, **`persist()` 즉시 `INSERT` 쿼리를 날려 ID를 받아옵니다.**

### **C. `flush()` vs `commit()`의 저장 메커니즘**

* **`flush()`:** "명령서 전달". 지금까지 쌓인 SQL 쿼리를 DB 서버로 전송만 합니다. DB 메모리에는 올라가지만, 아직 확정되지 않아 언제든 취소 가능합니다.
* **`commit()`:** "확정 도장". DB에 물리적 저장을 명령합니다.
* **테스트 환경:** `@Test` 환경에서는 `commit()` 직후 자동으로 **Rollback**이 수행되므로 로그에는 쿼리가 보여도 실제 DB 파일에는 데이터가 남지 않는 것이 정상입니다.

#### Q1. 테스트에서 commit해도 저장이 안 돼?
- 답변: 네, 일반적인 스프링 부트 환경의 @Test에서는 로그에 INSERT가 찍히더라도 실제 DB에는 남지 않는 것이 기본 설정입니다.
- 이유: 테스트 코드에 @Transactional이 붙어 있으면, 스프링은 테스트가 끝난 후 데이터가 DB에 남아서 다음 테스트에 영향을 주는 것을 막기 위해 무조건 롤백(Rollback)을 수행합니다.
- 해결: 실제로 DB에 들어가는 걸 눈으로 보고 싶다면 테스트 메서드 위에 @Rollback(false) 또는 @Commit 어노테이션을 붙여야 합니다.

## 3. 연관관계 어노테이션과 구조

### **`@ManyToOne` & `@JoinColumn**`

* **`@ManyToOne`:** "나는 여러 개(`Many`), 너는 하나(`One`)"라는 논리적 관계를 정의합니다.
* **`@JoinColumn(name = "item_id")`:** 실제 DB 테이블에 `item_id`라는 외래 키(FK) 컬럼을 생성하여 두 테이블을 연결하는 물리적 고리입니다.
* **`@OneToMany`:**기본값(Default)은 이미 LAZY입니다. 상품 하나에 입찰 기록이 1,000개 있다고 할 때, 상품을 조회하자마자 1,000개의 입찰을 다 가져오는 건 낭비이기 때문입니다.
  이때는 가짜 객체가 아니라 가짜 컬렉션(PersistentBag)이라는 특별한 프록시를 사용합니다.
  item.getBids()를 호출해도 쿼리는 안 나가고, 그 리스트 안에서 데이터를 하나 꺼내거나 size()를 묻는 등 리스트의 알맹이를 건드릴 때 비로소 쿼리가 나갑니다.

## 4. 자바 문법 및 롬복(Lombok) 주의사항

### **생성자(Constructor) 에러 해결**

* **대소문자 일치:** 클래스 이름이 `Bid`라면 생성자도 반드시 `public Bid`여야 합니다. `public bid`처럼 소문자로 쓰면 자바는 이를 리턴 타입이 누락된 **잘못된 메서드**로 오해하여 `void`를 붙이라고 에러를 냅니다.
* **롬복과 수동 생성자의 조화:**
* `@NoArgsConstructor`: JPA가 내부적으로 사용할 빈 껍데기(기본 생성자)를 만듭니다.
* 수동 생성자: 우리가 비즈니스 로직에서 데이터를 채워 넣을 때 사용합니다. ID는 DB가 자동 생성하므로 보통 ID를 제외한 필드들만 인자로 받습니다.



### **`em.find(Class, ID)`**

* **규약:** `em.find()`는 항상 "어떤 타입(Class)의 데이터를, 어떤 식별자(ID)로 찾을 것인가"를 인자로 받습니다.
* ID가 아닌 조건(이름, 가격 등)으로 찾고 싶다면 JPQL이나 QueryDSL 같은 별도의 쿼리 도구를 사용해야 합니다.

## 5. 오늘의 로그 복습 (Self-Check)

오늘 본 로그에서 다음 흐름을 다시 떠올려 보세요.

1. `em.find(Bid.class, id)` 호출 → 입찰 정보만 `SELECT`. 상품은 **Proxy** 상태.
2. `findBid.getItem().getClass()` → 쿼리 없음. 타입 정보만 확인.
3. `findBid.getItem().getName()` → 상품 테이블 **SELECT**. 프록시가 실제 데이터로 채워짐(초기화).

**결론:** JPA는 최대한 **게으르게(Lazy)** 일하려고 노력합니다. 진짜 데이터가 필요한 그 순간까지 조회를 미룸으로써 성능을 최적화하는 것이 2일차 공부의 핵심입니다.
