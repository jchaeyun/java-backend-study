# JPA 성능 최적화: N+1 문제와 Fetch Join

## 1. N+1 문제의 본질적 이해

### **정의**

조회 쿼리 1번($1$)을 실행했을 때, 연관된 데이터를 가져오기 위해 추가로 $N$번의 조회 쿼리가 실행되는 현상을 말합니다.

### **발생 원리: "지연 로딩(Lazy Loading)의 역설"**

JPA는 성능 최적화를 위해 `LAZY` 전략을 사용하면 연관된 엔티티를 **프록시(Proxy)** 객체로 채워둡니다.

1. **1차 쿼리:** 루트 엔티티(예: 상품)를 조회합니다. 이때 연관 엔티티(예: 입찰)는 프록시 상태입니다.
2. **지연 로딩 작동:** 비즈니스 로직에서 루프를 돌며 연관 엔티티의 실제 데이터를 호출합니다.
3. **N차 쿼리:** 영속성 컨텍스트는 각 객체마다 "알맹이가 없네?"라고 판단하고, 루프를 돌 때마다 DB에 `SELECT` 쿼리를 하나씩 날립니다.

---

## 2. 일반 Join vs Fetch Join (결정적 차이)

많은 초보 개발자가 "SQL에서 조인을 했는데 왜 또 쿼리가 나가지?"라고 당황합니다. 그 이유는 **JPA의 영속화 범위**에 있습니다.

### **일반 Join (`JOIN`)**

* **SQL:** 조인을 수행하지만 `SELECT` 절에는 **루트 엔티티의 컬럼만** 포함합니다.
* **JPA:** DB에서는 조인이 일어났지만, 결과 데이터를 받아올 때 연관 엔티티 정보가 없으므로 여전히 **프록시**로 남겨둡니다.
* **결과:** N+1 문제가 **해결되지 않습니다.**

### **Fetch Join (`JOIN FETCH`)**

* **SQL:** 조인을 수행하고 `SELECT` 절에 **연관 엔티티의 모든 컬럼**을 포함합니다.
* **JPA:** 한 번의 쿼리로 루트 엔티티와 연관 엔티티 데이터를 모두 읽어와 **1차 캐시에 진짜 객체**를 채워 넣습니다.
* **결과:** 객체 그래프 탐색 시 추가 쿼리가 전혀 발생하지 않습니다. **($1+N \rightarrow 1$)**

---

## 3. 실무형 비교 분석표

| 비교 항목 | 일반 Join | Fetch Join |
| --- | --- | --- |
| **핵심 목적** | 쿼리 필터링 (조회 조건으로만 사용) | **성능 최적화** (객체 그래프 통째로 로딩) |
| **SELECT 절** | 루트 엔티티만 포함 | 루트 + 연관 엔티티 모두 포함 |
| **영속화 상태** | 연관 엔티티는 **프록시** 유지 | 연관 엔티티는 **실제 엔티티**로 로딩 |
| **N+1 해결** | 불가능 | **가능** |

## 4. Fetch Join 사용 시 반드시 지켜야 할 3가지 원칙

Fetch Join은 강력하지만, 잘못 쓰면 서비스 장애를 일으킬 수 있습니다.

### **① 일대다(1:N) 페이징(Pagination) 금지**

* **문제:** 일대다 조인을 하면 데이터가 뻥튀기(중복)됩니다. 이때 페이징(`setFirstResult`)을 쓰면 JPA는 "데이터가 중복되어 정확한 페이징이 불가능하다"고 판단하고, **모든 데이터를 메모리로 긁어와서 자바에서 페이징**을 시도합니다. (**Out of Memory 위험**)
* **해결:** 다대일(N:1) 쪽에서 페치 조인을 하거나, `@BatchSize`를 사용하여 해결합니다.

---
### 1. $1:N$ vs $N:1$ 기준 세우기 (누가 주인공인가?)

둘은 같은 관계를 **어느 쪽에서 보느냐**의 차이일 뿐입니다.

* **일대다($1:N$):** 상품 한 개에 입찰이 여러 개 달려있죠? 상품(`1`) 입장에서 입찰(`N`)을 바라보면 일대다입니다.
* `VintageItem` 클래스 안에 `List<Bid> bids`가 있는 경우입니다.


* **다대일($N:1$):** 여러 입찰이 하나의 상품을 가리키죠? 입찰(`N`) 입장에서 상품(`1`)을 바라보면 다대일입니다.
* `Bid` 클래스 안에 `VintageItem item`이 있는 경우입니다.



---

### 2. 왜 일대다($1:N$) 페치 조인은 페이징이 안 될까?

이게 가장 중요한 **"데이터 뻥튀기(중복)"** 문제입니다.

### **상황 가정**

1번 상품(Polo Knit)에 입찰이 3개(A, B, C) 들어왔다고 해봅시다.

1. **일반 조회:** 1번 상품 정보 **1줄**만 가져옵니다.
2. **일대다 페치 조인 시 DB 결과:**
* SQL에서 상품과 입찰을 `JOIN`하면 결과 테이블은 다음과 같이 나옵니다.
* Row 1: [1번 상품 정보 | A 입찰 정보]
* Row 2: [1번 상품 정보 | B 입찰 정보]
* Row 3: [1번 상품 정보 | C 입찰 정보]





### **문제 발생**

JPA는 **상품**을 기준으로 페이징(예: 한 페이지에 상품 10개)을 하고 싶어 합니다. 그런데 DB 결과는 상품 1개당 데이터가 3줄로 늘어나 버렸죠?

* JPA 입장: "어? 나는 상품 10개를 가져오라고 명령받았는데, DB가 준 데이터는 상품 1개당 3줄씩이네? 어디서부터 어디까지가 진짜 1번 상품이지?"
* 결과: JPA는 멘붕에 빠지고, 결국 "내가 알아서 할게!"라며 DB의 페이징 기능(LIMIT, OFFSET)을 포기하고 **DB의 모든 데이터를 메모리로 다 가져와서** 자바 코드로 억지로 10개를 끊어냅니다. (데이터가 100만 개면 서버가 터집니다.)

---

### 3. 왜 다대일($N:1$) 페치 조인은 페이징이 안전할까?

반대로 입찰(`Bid`)을 주인공으로 해서 상품을 페치 조인하면 데이터가 늘어나지 않습니다.

* 입찰 1개는 반드시 **단 하나의 상품**에만 속합니다.
* **다대일 페치 조인 시 DB 결과:**
* Row 1: [A 입찰 정보 | 1번 상품 정보]
* Row 2: [B 입찰 정보 | 1번 상품 정보]
* Row 3: [C 입찰 정보 | 1번 상품 정보]


* 결과: 입찰 데이터 줄 수가 그대로 유지됩니다. 데이터가 뻥튀기되지 않으니 JPA가 안심하고 DB의 페이징 기능을 그대로 사용할 수 있습니다.

---

### 4. 그럼 일대다 페이징은 어떻게 해결하나? (BatchSize)

상품 리스트를 보여주면서 입찰 개수도 같이 보여주고 싶은데 페이징도 해야 한다면, **페치 조인을 포기**하고 이 방법을 씁니다.

#### 해결책: `@BatchSize`

1. 상품만 일반 쿼리로 페이징해서 가져옵니다. (상품 10개 조회)
2. 입찰 내역은 `LAZY` 상태로 둡니다.
3. 설정된 `BatchSize`(예: 100)만큼 **연관된 입찰 데이터들을 `IN` 쿼리로 한꺼번에** 긁어옵니다.

* **작동 원리:** "1번 상품 입찰 가져와", "2번 상품 입찰 가져와"... 이렇게 $N$번 묻는 대신, "1, 2, 3...10번 상품에 해당하는 입찰들 한 번에 다 가져와!"라고 딱 1번만 묻는 것입니다.
* 결과적으로 쿼리가 $1+N$에서 $1+1$ 정도로 획기적으로 줄어들면서 페이징도 안전하게 가능해집니다.



### **요약**

* **일대다($1:N$):** 부모(1)가 자식(N)을 찾음 → 데이터가 N배로 늘어남 → **페치 조인 시 페이징 절대 금지!**
* **다대일($N:1$):** 자식(N)이 부모(1)를 찾음 → 데이터 줄 수 변화 없음 → **페치 조인 & 페이징 가능!**
* **해결사:** 일대다 관계에서 페이징이 꼭 필요하면 `@BatchSize`가 정답입니다.

---

### **② 둘 이상의 컬렉션 페치 조인 금지**

* **문제:** 컬렉션(List 등) 두 개를 동시에 페치 조인하면 데이터가 기하급수적으로 늘어나는 카테시안 곱(Cartesian Product)이 발생합니다.
* **결과:** 하이버네이트에서 `MultipleBagFetchException`을 던지며 실행을 거절합니다.

### **③ 엔티티 그래프는 가급적 단순하게**

* 너무 많은 단계를 한꺼번에 페치 조인하면(예: A -> B -> C -> D) SQL이 복잡해지고 인덱스 활용이 어려워질 수 있습니다. 꼭 필요한 경로만 최적화합니다.

## 5. 실습 코드 요약 

### **[N+1 발생 상황]**

```java
// 모든 상품을 가져오지만 입찰 내역은 여전히 프록시
List<VintageItem> items = em.createQuery("select v from VintageItem v", VintageItem.class)
                            .getResultList();

for (VintageItem item : items) {
    // 프록시 강제 초기화 시점에 N번의 추가 쿼리 발생!
    System.out.println(item.getBids().size());
}

```

### **[Fetch Join 적용]**

```java
// 한 번의 쿼리로 연관된 입찰 내역까지 모두 실제 객체로 조회
String jpql = "select v from VintageItem v join fetch v.bids";
List<VintageItem> items = em.createQuery(jpql, VintageItem.class)
                            .getResultList();

for (VintageItem item : items) {
    // 이미 1차 캐시에 '진짜 데이터'가 있어 쿼리 발생 0회
    System.out.println(item.getBids().size());
}

```
