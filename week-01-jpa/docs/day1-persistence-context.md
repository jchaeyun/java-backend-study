# JPA 핵심 동작 원리와 실무 적용 개념 정리

## 1. 영속성 컨텍스트 (Persistence Context) 내부 구조

애플리케이션과 데이터베이스 사이의 논리적 작업 공간(메모리)으로, 데이터베이스 부하를 줄이고 객체 상태를 관리한다.

* **1차 캐시 (Entity 저장소):**
* `Key`(@Id)와 `Value`(엔티티 객체)를 매핑하여 저장하는 Map 형태의 공간.
* 단일 스레드의 트랜잭션과 생명주기를 같이 하는 로컬 캐시.


* **스냅샷 (Snapshot):**
* 1차 캐시 내부에 보관됨.
* DB에서 데이터를 최초로 읽어와 1차 캐시에 저장할 때의 초기 상태를 복사해 둔 데이터. 변경 감지(Dirty Checking)의 비교 기준이 됨.


* **쓰기 지연 SQL 저장소 (ActionQueue):**
* 데이터(엔티티)가 아닌 DB에 보낼 명령(SQL 쿼리 자체)을 순서대로 모아두는 대기열(Queue).



## 2. 데이터 동기화 단계별 메서드 구분

데이터베이스 트랜잭션 메커니즘에 기반하여 각 단계는 명확히 분리된다.

* **`persist()` (메모리 작업):**
* 엔티티를 1차 캐시에 등록하고, `ActionQueue`에 `INSERT` 쿼리를 생성하여 대기열에 올림.
* DB와 네트워크 통신을 하지 않음.


* **`flush()` (네트워크 작업):**
* `ActionQueue`에 대기 중인 SQL 쿼리들을 DB 서버로 전송.
* 영구 저장이 아님. DB 메모리 상에 임시로 쿼리를 실행해 둔 상태이며, 에러 발생 시 롤백됨.


* **`commit()` (영구 저장 작업):**
* 내부적으로 `flush()`를 선행 호출하여 남은 쿼리를 전송.
* DB에 물리적 디스크 기록(영구 저장)을 명령하는 `COMMIT` 신호를 하달함.



## 3. 쓰기 지연의 실무적 한계와 JDBC Batch

`ActionQueue`에 쿼리를 모아두는 것만으로는 네트워크 병목을 완벽히 해결할 수 없다.

* **동작 원리와 한계:**
* 기본 설정 상태에서는 쿼리가 모여 있어도 `flush()` 시점에 네트워크 패킷을 건건이 따로 전송함.


* **JDBC Batch 비활성화 이유:**
* DB 벤더별 드라이버 지원 방식이 다르고, 잘못 설정 시 메모리 누수가 발생할 수 있어 기본값은 비활성화 상태.


* **해결책 (application.yml 설정):**
* 여러 쿼리를 하나의 패킷으로 묶어 전송하도록 명시적 활성화 필요.


```yaml
spring:
  jpa:
    properties:
      hibernate:
        jdbc.batch_size: 50
        order_inserts: true
        order_updates: true

```

* `GenerationType.IDENTITY` 키 생성 전략 사용 시, `INSERT` 즉시 ID 값을 알아야 하므로 Batch 처리가 무효화됨.
**`IDENTITY` 전략(주로 MySQL, H2에서 사용)을 쓰면 `persist()`가 호출되는 즉시 `INSERT` 쿼리가 실행됨.**

원래 JPA의 대원칙은 "트랜잭션이 끝날 때(`commit`) 혹은 수동으로 `flush`를 할 때 쿼리를 한꺼번에 보낸다"이지만, `IDENTITY`는 이 원칙을 깨뜨리는 **유일한 예외**.

---

## ## 왜 IDENTITY는 즉시 쿼리를 보낼까?

이유는 **"1차 캐시의 구조"** 때문.

1. **Map 구조:** 영속성 컨텍스트(1차 캐시)는 내부적으로 `Map` 형태. `Map`에 데이터를 넣으려면 반드시 Key(ID)와 Value(엔티티)가 세트로 있어야 함.
2. **번호표가 없음:** `IDENTITY` 전략은 "일단 DB에 데이터를 넣어봐야" DB가 자동으로 번호(ID)를 매겨줌. 즉, 넣기 전까지는 자바 객체 입장에서 본인의 ID가 뭔지 알 방법이 없음.
3. **JPA의 선택:** 캐시에 저장(Key-Value 매핑)은 해야 하는데 Key(ID)가 없으니, JPA는 **어쩔 수 없이 즉시 DB에 `INSERT`를 날려서 번호표(ID)를 받아온 뒤** 캐시에 등록하는 방식을 택한 것.

---

## ## 이로 인해 발생하는 현상

* **쓰기 지연(Write-behind) 불가능:** 쿼리를 모았다가 한꺼번에 쏘는 '쓰기 지연' 기능이 이 브랜치에서는 작동하지 않음. `persist()`를 10번 호출하면 쿼리도 즉시 10번 나감.
* **JDBC Batch Insert 최적화 불가:** 여러 개의 데이터를 한 번에 넣는 최적화 기법도 `IDENTITY` 전략에서는 사용할 수 없습니다. 하나를 넣어야 그다음 ID를 알 수 있기 때문.

---

## ## 요약하자면

* **일반적인 경우 (SEQUENCE 등):** `persist` (캐시 저장) -> `commit/flush` (쿼리 전송)
* **IDENTITY의 경우:** **`persist` (즉시 쿼리 전송 + ID 획득 + 캐시 저장)** -> `commit/flush` (이미 쿼리 나갔음)

그래서 아까 로그에서 `===persist 호출 전===`과 `===persist 호출 후===` 사이에 쿼리가 찍혔던 것. `IDENTITY` 전략을 사용하고 있다면 이는 지극히 정상적인 동작.






## 4. 동시성 문제와 데이터 정합성 보장

동시 다발적인 트랜잭션 발생 시(예: 경매 입찰), 1차 캐시의 고립성으로 인해 구형 데이터(Stale Data)를 읽는 문제가 발생한다. 1차 캐시를 비활성화하는 방식으로는 해결할 수 없으며, 락(Lock) 메커니즘을 적용해야 한다.

* **비관적 락 (Pessimistic Lock):**
* 데이터 조회 시점에 DB의 해당 Row 자체에 배타적 자물쇠를 채움 (`SELECT ... FOR UPDATE`).
* 다른 트랜잭션의 접근 자체를 대기시킴.


* **낙관적 락 (Optimistic Lock):**
* 엔티티에 `@Version` 필드를 두어 애플리케이션 레벨에서 버전을 관리.
* 수정 시점에 버전 충돌이 발생하면 예외를 던지고 재시도 로직을 수행함.



## 5. 엔티티 설계 규약 및 Lombok 활용

* **기본 생성자 규약 (`protected`):**
* JPA(Hibernate)는 데이터 조회 시 리플렉션(Reflection)과 프록시(Proxy)를 통해 객체를 동적 생성하므로 파라미터가 없는 기본 생성자가 필수적임.
* `public`으로 개방할 경우 의미 없는 빈 객체가 무분별하게 생성되어 캡슐화가 훼손되므로 `protected`로 제한해야 함.


* **Lombok 적용:**
* `@NoArgsConstructor(access = AccessLevel.PROTECTED)`를 사용하여 보일러플레이트 코드를 줄이면서 객체 무결성을 보호함.



### 1. "의미 없는 빈 객체의 무분별한 생성"은 왜 문제일까?

우리가 만든 `CheerMessage` 엔티티가 `내용(content)`과 `작성자(writer)`가 반드시 있어야 한다고 가정해 봅시다.

* **Public으로 열려 있을 때:**
  누군가 코드 어디선가 `new CheerMessage();`라고 호출해서 아무 내용도 없는 빈 껍데기 객체를 만들 수 있습니다. 이렇게 만들어진 객체는 비즈니스 로직을 수행할 때 에러를 일으키는 '시한폭탄'이 됩니다.
* **캡슐화 훼손:**
  객체는 스스로의 무결성(필수 데이터가 다 들어있는지)을 책임져야 합니다. 아무나 빈 객체를 만들 수 있게 허용하는 순간, 그 책임이 무너지는 것이죠.

---

### 2. 그런데 왜 아예 막지 못하고 `protected`로 열어둘까?

여기서 **JPA(Hibernate)의 특성**이 등장합니다. JPA는 우리가 만든 엔티티를 관리할 때 내부적으로 '프록시(Proxy)'라는 가짜 객체를 만들어서 사용합니다.

* **JPA의 요구사항:** JPA는 DB에서 데이터를 가져와 객체에 채워넣을 때, 일단 빈 객체를 생성한 뒤에 데이터를 주입합니다. 이때 기본 생성자(No-args constructor)가 반드시 필요합니다.
* **협상 지점:**
* `public`: 아무나 빈 객체를 만들 수 있어서 보안상 불안함.
* `private`: JPA가 이 객체를 생성하지 못해서 에러가 발생함.
* **`protected`**: **"일반 사용자(외부 패키지)는 빈 객체를 못 만들게 막으면서, JPA 프레임워크는 접근할 수 있게 열어두는"** 가장 적절한 타협점입니다.



---

### 3. Lombok의 `@NoArgsConstructor(access = AccessLevel.PROTECTED)`

이 긴 설정을 매번 코드로 치기 귀찮으니 **Lombok**이 대신 해주는 것입니다.

#### 코드로 비교하기

**기존 방식 (직접 작성):**

```java
@Entity
public class CheerMessage {
    // ... 필드들 ...

    // JPA를 위해 억지로 만들었지만, 외부에서는 못 쓰게 protected로 제한
    protected CheerMessage() {
    }
}

```

**Lombok 방식 (어노테이션 한 줄):**

```java
@Entity
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class CheerMessage {
    // ... 필드들 ...
}

```
